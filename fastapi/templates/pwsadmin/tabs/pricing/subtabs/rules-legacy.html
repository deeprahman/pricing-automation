Simple Help: build a rule without JSON

Simple mode is for beginners. Fill in normal form fields, choose the rule target scope, and the dashboard will build the JSON rule in the background for you.

What Simple mode supports

  • Core pricing actions only: increase, decrease, set, multiplier
  • Date range rules and exact-date rules
  • Optional longer-stay conditions using stay length and booking classes
  • Scope-aware targets: property + platform, property only, platform only, global, or one linked listing
  • Beginner-friendly create and edit for compatible rules

How to fill it in

  1. 1. Pick the target scope first.
  2. 2. Choose what pricing action to apply.
  3. 3. Choose which booking categories should trigger it.
  4. 4. Add a stay-length condition when the rule is for longer stays.
  5. 5. Choose either a date range or exact dates.
  6. 6. Save the rule.

Editing existing rules

  • Click a row in the pricing table to load it into the editor.
  • If a rule uses extra JSON-only features, the dashboard will switch you to Advanced mode automatically.
  • Use New Rule to leave edit mode and create a new item.

Longer-stay rules

Turn on Longer-stay condition when the rule should match by booking length. Simple mode writes this in v2 format with rule_config.conditions_version = 2 and a rule_config.condition_tree that includes stay_length plus category/class matching. For execution, the pricing database function must include p_stay_length and p_booking_classes.

How rule selection changes

  • Longer-stay off + trigger categories present: the worker selects the rule when one incoming category matches the condition_tree booking category any_of node.
  • Longer-stay on + trigger categories present: the worker selects the rule only when both nodes in the condition_tree AND group pass: stay_length and booking class/category matching.
  • No longer-stay and no trigger categories: the structured worker has no booking signal to match, so the rule is not selected for instruction creation.
  • Stay length without booking classes: possible only in Advanced JSON; it can match any booking with that length, so use it carefully.

What the selected rule creates

After selection, the pricing worker forwards the rule, matched category, target listing, and booking dates to the BSO worker. The BSO worker uses operation, apply_window, and the booking arrival/departure to create one external-service instruction per matched linked listing.

Beginner example

Property = Summa St Cottage
Platform = airbnb
Action = increase
Type = percentage
Amount = 12
Apply from = departure
Duration days = 2
Offset days = 0
Trigger categories = checkout, possible_stay_extension
Longer-stay condition = stay length greater than 15 nights
Date scope = date range
Start date = 2026-01-01
End date = 2026-12-31
Priority = 90
Status = active

Meaning: when one of the selected categories matches for that target, raise the price by 12% for 2 days starting from departure.

Moderate Help: more rule types without JSON

Moderate mode keeps the beginner-friendly form style, but adds more structured pricing tools like price floors, stay-adjustment conditions, availability controls, and day-of-week schedules.

What Moderate adds

  • Pricing controls: min_price, max_price, remove_overrides
  • Availability controls: close_dates, open_dates
  • Length-of-stay controls: min_stay, max_stay
  • Stay-adjustment filters: stay_length.gte/lte, stay_extended, stay_contracted, net_stay.gte/lte
  • Rate baselines with operation.target_rate_type: base, recommended, minimum, or maximum
  • Season windows with year-agnostic MM-DD ranges, including wrapping ranges like 12-10 to 03-21
  • Weekly schedules with day_of_week_pattern

How to use it

  1. 1. Pick the target scope and then fill the matching selector fields.
  2. 2. Choose the structured operation you want.
  3. 3. Fill only the fields shown for that operation.
  4. 4. Pick trigger categories, optional stay-adjustment criteria, and the scope.
  5. 5. Save without touching raw JSON.

Examples

  • min_price: never let selected dates drop below 140
  • close_dates: block all selected dates from booking
  • min_stay: require at least 3 nights
  • stay_length.gte/lte: constrain original stay-length ranges
  • stay_extended / stay_contracted: exact extension/contraction match
  • net_stay.gte/lte: constrain final net stay window
  • target_rate_type: choose which stored rate snapshot is the baseline; default is base
  • season_window: match dates by month/day without choosing a year
  • day_of_week_pattern: run every Friday and Saturday only

When it switches to Advanced

If a rule contains extra JSON branches or a shape Moderate cannot safely rebuild, the dashboard will move you to Advanced mode automatically so nothing is lost.

Advanced Help: how this works + what to enter

In this project, a pricing rule tells the SOA/BSO pipeline what to do when a booking matches a category such as potential_extension. The pricing engine now resolves rules in this order: listing -> property -> platform -> global.

How the flow works

  1. 1. A booking is categorized, for example potential_extension.
  2. 2. The pricing worker resolves active rules from the most specific scope down to the broadest one.
  3. 3. If multiple rules match, the higher priority wins.
  4. 4. rule_config supplies the category match, operation, and date window used to build the instruction.

Filter fields above

  • scope: narrow the list to listing, property, platform, or global
  • property_id: show rules for one property. Example: 389550
  • platform_id: narrow to one platform. Example: 432 for Airbnb in the demo
  • lookup_id: show rules for one linked listing via platform_property_lookup_id
  • operation_code: find one action type. Example: increase
  • status: use active to find live rules, or inactive to find saved-but-disabled rules

Important tips

  • Listing-scoped rules use platform_property_lookup_id and must leave property_id and platform_id blank.
  • Property rules may keep platform_id blank or optionally narrow the property to one platform.
  • You must give the rule a date scope: exact applicable_dates, a start_date + end_date range, or day_of_week_pattern in Moderate/Advanced mode.
  • Primary trigger format: set rule_config.conditions_version to 2 and define matching logic in rule_config.condition_tree.
  • Legacy note: some older rules may still use rule_config.conditions... or top-level stay keys. They can still appear and run, but new authoring should use v2 condition_tree.

Field-by-field examples

  • rule_uuid: leave empty when creating a new rule; fill it when updating an existing rule. Example: 3fa85f64-5717-4562-b3fc-2c963f66afa6
  • operation_code: the registered action to apply. Demo example: increase
  • property_id: the property the rule belongs to. Leave it blank for platform, global, or listing scope. Demo example: 389550
  • platform_id: the target platform for that rule. Leave it blank for global rules, and also blank for listing scope. Demo example: 432
  • platform_property_lookup_id: the linked listing target for a listing-scoped rule. Example: 42
  • priority: higher number wins when rules overlap. Demo example: 90
  • status: active means the worker can use it; inactive means keep it saved but ignored.
  • rule_config: JSON that defines the subject, operation, optional apply window, and v2 trigger logic (conditions_version + condition_tree).
  • applicable_dates: use this when the rule should hit exact dates only. Example: ["2026-02-23","2026-02-24"]
  • start_date: first day of a date-range rule. Example: 2026-01-01
  • end_date: last day of a date-range rule. Example: 2026-12-31

Condition Tree v2 reference

Use this as the primary format for new Advanced rules.

  • Required root keys: subject, operation, conditions_version = 2, condition_tree.
  • Optional root keys: apply_window and other metadata your pipeline already supports.
  • Group node shape: {"type":"group","evaluation_operator":"and|or","members":[...]}
  • Condition node shape: {"type":"condition","condition_name":"...","comparison_operator":"...","value":...}
  • Optional pos is supported for booking_category/booking_class conditions when position-aware matching is needed.
Familycondition_namecomparison_operatorvalue format
Numeric stay_length, stay_extended, stay_contracted, net_stay eq, gt, gte, lt, lte, between number, or {"min":n,"max":n} for between
Category / class booking_category, booking_class any_of, all_of non-empty string array
Date arrival_date, departure_date, target_date eq, gt, gte, lt, lte, between ISO date string, or {"min":"YYYY-MM-DD","max":"YYYY-MM-DD"} for between

For between, min cannot be greater than max.

Copy/paste examples (condition_tree v2)

1. Minimal single-condition rule

rule_uuid = leave empty when creating
operation_code = increase
property_id = 389550
platform_id = 432
priority = 90
status = active
rule_config = {
  "subject": "price",
  "operation": { "type": "percentage", "amount": 12 },
  "apply_window": { "applies_from": "departure", "duration_days": 2, "offset_days": 0 },
  "conditions_version": 2,
  "condition_tree": {
    "type": "condition",
    "condition_name": "booking_category",
    "comparison_operator": "any_of",
    "value": ["potential_extension", "extension"]
  }
}
applicable_dates = leave empty
start_date = 2026-01-01
end_date = 2026-12-31

2. Nested AND/OR group rule

rule_uuid = leave empty when creating
operation_code = increase
property_id = 389550
platform_id = 432
priority = 90
status = active
rule_config = {
  "subject": "price",
  "operation": { "type": "percentage", "amount": 18 },
  "apply_window": { "applies_from": "departure", "duration_days": 3, "offset_days": 0 },
  "conditions_version": 2,
  "condition_tree": {
    "type": "group",
    "evaluation_operator": "and",
    "members": [
      {
        "type": "group",
        "evaluation_operator": "or",
        "members": [
          { "type": "condition", "condition_name": "stay_length", "comparison_operator": "between", "value": { "min": 10, "max": 20 } },
          { "type": "condition", "condition_name": "stay_extended", "comparison_operator": "gt", "value": 0 }
        ]
      },
      {
        "type": "condition",
        "condition_name": "booking_class",
        "comparison_operator": "any_of",
        "value": ["medical_related", "job_related"],
        "pos": [0, null]
      }
    ]
  }
}
applicable_dates = leave empty
start_date = 2026-01-01
end_date = 2026-12-31

Instruction creation examples

1. v2 booking-category price increase

Rule:
conditions_version = 2
condition_tree = booking_category any_of ["checkout"]
operation = increase
type = percentage
amount = 12
apply_window = departure + 2 days

Booking:
departure = 2026-02-23
categories = ["checkout"]

Instruction:
operation = increase
type = percentage
amount = 12
trigger = condition_tree booking_category any_of
dates = ["2026-02-23","2026-02-24"]

2. v2 stay + class AND rule

Rule:
conditions_version = 2
condition_tree = AND(
  booking_class any_of ["job_related"],
  stay_length gte 15,
  stay_extended eq 2
)
operation = increase
type = percentage
amount = 40
apply_window = departure + offset 2, duration 3

Booking:
stay_length = 45
stay_extended = 2
stay_contracted = 0
booking_classes = ["job_related"]
departure = 2026-04-15

Instruction:
operation = increase
type = percentage
amount = 40
trigger = condition_tree AND group
dates = ["2026-04-17","2026-04-18","2026-04-19"]

3. v2 listing-scoped fixed price rule

Rule:
scope = listing
conditions_version = 2
condition_tree = booking_category any_of ["possible_stay_extension"]
operation = set
type = fixed
amount = 175
apply_window = arrival before 1 day

Booking:
arrival = 2026-03-10
categories = ["possible_stay_extension"]

Instruction:
operation = set
type = fixed
amount = 175
trigger = condition_tree booking_category any_of
dates = ["2026-03-09"]
